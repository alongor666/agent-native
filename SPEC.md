# Agent-Native Specification (ANS)

> **Agent-Native Specification** · 中文名《Agent-Native 规范》
> 版本 0.2.0 · 候选(Candidate) · 2026-06-07 · 技术栈无关 · 领域无关
> 授权 [CC-BY-4.0](./LICENSE) · 演进规则见 [GOVERNANCE.md](./GOVERNANCE.md) · 机器可读契约 [`ans.json`](./ans.json)

## 摘要

当软件的**开发者与消费者都是 AI Agent**而非人类，其「接口面」必须为 Agent 的认知特性重建。
本规范定义四条公理——**确定性契约、可验证闭环、可恢复状态、安全边界**——与一致性验收方法，
适用于任何软件产物（库 / 服务 / 前端 / CLI / 数据 / 基础设施），不绑定任何语言、框架或领域。

本文档中关键词「**必须**(MUST)」「**应当**(SHOULD)」「**可以**(MAY)」依 RFC 2119 解释。

---

## 0. 动机

人类与 Agent 的认知差异是**固定的**，软件接口若为人类优化，对 Agent 即为歧义与失明之源：

| | 人类 | AI Agent | 推导出的公理 |
| --- | --- | --- | --- |
| 记忆 | 持久、有经验 | 上下文有限且易失 | ③ 可恢复状态 |
| 歧义 | 直觉脑补 | 要么有契约，要么幻觉 | ① 确定性契约 |
| 行动 | 直接操作 | 经工具行动，需反馈才知对错 | ② 可验证闭环 |
| 试错 | 成本高、串行 | 成本低、可并行；**不可逆操作例外** | ④ 安全边界 |
| 阅读 | 叙事 | 结构 | ① 确定性契约 |

Agent-native 不是某类项目的模板，而是这组差异推导出的、跨领域的设计公理。

## 1. 范围

适用于任何被 Agent 开发或消费的软件产物，与具体架构正交。一个项目可只满足部分公理；
满足度即其 **agent-native 成熟度**（见 §3）。本规范描述**做什么**与**为何**，不规定实现技术。

## 2. 四公理（规范性）

### 公理一 · 确定性契约 — Determinism over Inference

**定义**：凡需 Agent 推断之处皆为幻觉源。一切接口、数据、版本、能力**必须**机器可判定、自描述、单一真源。

**规范要求**
- 接口契约 **MUST** 机器可读（类型签名 / JSON Schema / OpenAPI / IDL），不得**仅**以散文表达。
- 同一事实 **MUST** 单一真源；派生物自动生成，并由自动校验锁定一致（防漂移）。
- 版本兼容性 **MUST** 机器可判定（语义化版本 + 显式兼容语义：加 = 兼容 / 改 = 破坏）。
- 项目能力与入口 **SHOULD** 有机器可读的发现清单，Agent 无需读散文或猜目录结构。
- 面向人的叙事文档与面向 Agent 的契约 **SHOULD** 分离（各自最优）。
- 约束 Agent 行为的规则 **SHOULD** 分层：可被工具机械执行 / 拒止的**强制配置** 与 仅供理解的**参考叙事** 分开；硬约束不得仅以散文承担。

**反模式**：散文即契约、隐式默认、魔法常量、靠经验的约定俗成。

**正例**：能力发现握手 —— 服务端在连接建立时以机器可读清单声明可调用的能力集合，消费方只调用清单中已协商者（如 MCP 的 `initialize` + `tools`/`resources`/`prompts` 发现）。
**反例**：README 驱动集成 —— 参数格式、版本兼容、认证方式只写在散文文档里，对接方须人工转译、无法程序化验证自己是否兼容。

### 公理二 · 可验证闭环 — Closed-Loop Verifiability

**定义**：Agent 经工具行动，**必须**从行动获得机器可读的结果与失败原因，方能自我纠错。沉默的成功与散文的失败 = Agent 失明。

**规范要求**
- 每个操作 **MUST** 返回机器可读结果（结构化输出 + 语义化退出码 / 状态）。
- 失败 **MUST** 机器可分类（错误码 / 类型），不得**仅**为自然语言消息。
- 失败分类 **SHOULD** 含可操作维度（错误域 + 是否可重试），使调用方能程序化决定「重试 / 改参 / 放弃」，而非仅得一个布尔。
- 产物 **SHOULD** 携带 provenance（输入指纹 + 代码版本 + 执行者 + 时间），可被另一 Agent 独立复现与验证。
- 构建与产出 **SHOULD** 可复现（确定性）。

**反模式**：成功静默、失败 print 散文、副作用不可观测、产出不可复现。

**正例**：双层失败报告 —— 「工具执行失败」作为结构化结果返回（模型可见、可据此纠错），「调用 / 协议层失败」用标准错误码（参数非法等）；调用方据此机械区分该重试还是该改参（如 MCP 的 `isError` + JSON-RPC 错误码）。
**反例**：沉默失败 —— 某步骤失败却不置退出码，或只 `print("出错了")`，调用方无法分类、只能人读日志。

### 公理三 · 可恢复状态 — Resumable State

**定义**：Agent 上下文易失、长任务会中断。状态**必须**外置、可重入。

**规范要求**
- 长流程 **MUST** 幂等或可断点续跑；中断后重跑不得产生重复副作用或损坏数据。
- 状态 **MUST** 外置于持久载体，不得**仅**存于 Agent 上下文或隐藏的全局可变态。
- 运行期施加的**约束与规则**亦属状态：**MUST** 外置于持久载体，不得仅以会话内对话承载（上下文压缩 / 截断会使其无声失效）。
- 操作 **SHOULD** 幂等（幂等键 / 期望状态收敛）。

**反模式**：一次性不可重入脚本、隐藏可变状态、"挂了从头来"。

**正例**：状态外置 + 不可变分叉 —— 会话历史与进度持久化为外部文件，冷启动可重放续跑；从历史节点分叉时新建标识、原状态保持不可变，失败可回到原点重试。
**反例**：30 步脚本在第 28 步失败便从头重跑，而其间的迁移 / 上传等已造成不可回滚副作用，重跑引发重复或冲突。

### 公理四 · 安全边界 — Safe-by-Default Boundaries

**定义**：Agent 行动快且会错。可逆操作可自由试错；**不可逆 / 外发 / 破坏性操作必须设界**。

**规范要求**
- 不可逆 / 外发 / 破坏性操作 **MUST** 显式确认或具备回滚，不得为静默默认。
- 硬性安全边界 **MUST** 由**外部机制强制**（沙箱 / 权限系统 / 拒止规则 / 不可变审计），可独立于 Agent 的判断生效；基于 AI 或启发式的自检属 best-effort，**MUST NOT** 作为不可逆损害的唯一防线。
- 操作 **SHOULD** 提供 dry-run / preview。
- Agent 运行 **SHOULD** 遵循最小权限，且变更可审计、可回滚。

**反模式**：破坏性默认、无 dry-run、无回滚、宽权限、不可审计、把 AI 自检当唯一防护。

**正例**：内核级沙箱 + 计划预览 —— 边界在 OS / 权限层强制，Agent 即便被提示注入也越不过；不可逆操作先进入只读「计划 / 预览」态，须显式批准方可执行。
**反例**：开发工具默认直连生产、无 dry-run，一次误操作不可逆清空生产数据，安全全靠人工纪律。

## 3. 一致性（Conformance）

### 3.1 四问验收（机械可执行）

对任意项目，逐条回答；全为「是」即符合本规范：

1. **契约**：一个从未见过它的 Agent，能否**只靠机器可读文件**（不读散文、不问人）正确调用 / 消费 / 扩展它？
2. **反馈**：执行任一操作后，能否**程序化判断**成功 / 失败 / 失败原因？
3. **恢复**：任一长流程中断后重跑，是否**安全且不重复副作用**？
4. **安全**：Agent 能否在**不造成不可逆损害**的前提下自由试错？

### 3.2 成熟度分级

| 级别 | 名称 | 条件 | Agent 可做到 |
| --- | --- | --- | --- |
| **L0** | 人类原生 | 散文契约、人读日志、不可逆默认 | 需人类在环翻译 |
| **L1** | 契约就绪 | 满足公理一 | 正确消费 / 扩展，无需猜 |
| **L2** | 闭环就绪 | + 公理二、三 | 自我纠错、中断可恢复 |
| **L3** | 自治就绪 | + 公理四 | 在安全边界内自主开发与运维 |

## 4. 与现有工作的关系

本规范不发明新机制，而是把已有实践统一进四公理坐标系。下表与逐条说明把**每个标准精确投影到它主要满足的那条 MUST/SHOULD**（引用 §2 原文关键词），并标注它**不覆盖**哪些公理——以此坐实：**四公理互相独立，无任何单一标准全包**；达 L3 必须组合多个标准。

> 引用记法：`①-MUST/接口契约机器可读` 指公理一中以「接口契约 MUST 机器可读」开头的那条。关键词与 §2 逐字对齐，可程序化交叉校验。

### 4.1 标准 × 命中的 MUST/SHOULD 对照表

| 标准 | 主投影公理 | 命中的具体条目（关键词） | 等级 | 不覆盖 |
| --- | --- | --- | --- | --- |
| **MCP** (Model Context Protocol) | ① + ② | ①「能力与入口 SHOULD 有机器可读发现清单」；②「每操作 MUST 返回机器可读结果」「失败 MUST 机器可分类」 | SHOULD+MUST | ③ ④ |
| **OpenAPI** | ① | ①「接口契约 MUST 机器可读（…OpenAPI…）」「版本兼容性 MUST 机器可判定」 | MUST | ②(部分) ③ ④ |
| **llms.txt** | ① | ①「能力与入口 SHOULD 有机器可读发现清单」「人读叙事与 Agent 契约 SHOULD 分离」 | SHOULD | ② ③ ④ |
| **AGENTS.md / CLAUDE.md** | ① | ①「约束 Agent 行为的规则 SHOULD 分层：强制配置 vs 参考叙事」「叙事与契约 SHOULD 分离」 | SHOULD | ② ③ ④ |
| **声明式 IaC**（Terraform 等） | ③ + ① + ④ | ③「长流程 MUST 幂等或可断点续跑」「操作 SHOULD 幂等」「状态 MUST 外置于持久载体」；①「声明式期望状态」；④「不可逆操作 SHOULD 提供 dry-run/preview」「MUST 显式确认或回滚」 | MUST+SHOULD | ②(细粒度反馈) |
| **SemVer** | ① | ①「版本兼容性 MUST 机器可判定（语义化版本 + 显式兼容语义：加=兼容/改=破坏）」 | MUST | ② ③ ④ |
| **OpenTelemetry / 结构化日志** | ② | ②「每操作 MUST 返回机器可读结果」「失败 MUST 机器可分类」「失败分类 SHOULD 含错误域+可重试」「产物 SHOULD 携带 provenance」 | MUST+SHOULD | ① ③ ④ |
| **SLSA provenance** | ② | ②「产物 SHOULD 携带 provenance（输入指纹+代码版本+执行者+时间）」「构建与产出 SHOULD 可复现」 | SHOULD | ① ③ ④ |

### 4.2 逐标准映射

- **MCP** —— 主投影 **①②**。`initialize` 握手协商协议版本与能力集，`tools/list`·`resources/list`·`prompts/list` 返回机器可读的能力清单，直接满足 ①「能力与入口 SHOULD 有机器可读发现清单」；工具调用结果是结构化 `content`，工具级失败置 `isError:true` 由模型可见可纠错、协议级失败用 JSON-RPC 标准错误码（如 `-32602` 参数非法），恰好实现 ②「每操作 MUST 返回机器可读结果」+「失败 MUST 机器可分类」的双层失败语义。**不覆盖 ③④**：协议本身不规定状态外置/断点续跑，也不强制任何安全边界或回滚——破坏性工具是否设界由实现自负。

- **OpenAPI** —— 主投影 **①**。以 JSON Schema 描述请求/响应/参数，是 ①「接口契约 MUST 机器可读（类型签名 / JSON Schema / OpenAPI / IDL）」的点名机制；`info.version` + 可选 `responses` 错误响应 schema（如 RFC 9457 problem+json）部分触及 ②「失败 MUST 机器可分类」。**不覆盖**：它**声明**契约但不**强制运行期反馈**（真实返回是否符 schema 需另测），故 ② 仅部分命中；与 ③④ 正交。

- **llms.txt** —— 主投影 **①**。站点根放置机器可发现的 Markdown 入口清单，让 Agent 无需爬全站或猜目录即可定位关键资源，命中 ①「能力与入口 SHOULD 有机器可读发现清单」+「人读叙事与 Agent 契约 SHOULD 分离」。**不覆盖 ②③④**：它只是发现层指引，不提供调用反馈、状态或边界。

- **AGENTS.md / CLAUDE.md** —— 主投影 **①**。把面向 Agent 的工程约定（构建命令、风格、禁区）从面向人的 README 中析出，命中 ①「叙事与契约 SHOULD 分离」+「约束 Agent 的规则 SHOULD 分层」。**注意其等级上限**：作为散文文件，它只能承载「参考叙事」与 SHOULD；①明确「硬约束不得仅以散文承担」，故强制规则仍须落到工具可机械拒止的配置（lint/hook/权限），AGENTS.md 不能独力满足硬约束。**不覆盖 ②③④**。

- **声明式 IaC（Terraform 等）** —— 主投影 **③**，兼及 **①④**。期望状态声明（HCL/JSON）是 ①「声明式期望状态」；`apply` 收敛、可重复执行不产生重复副作用，命中 ③「长流程 MUST 幂等或可断点续跑」+「操作 SHOULD 幂等」，state 文件外置命中 ③「状态 MUST 外置于持久载体」；`plan` 预览 + 审批命中 ④「SHOULD 提供 dry-run/preview」+「MUST 显式确认或回滚」。**不覆盖 ②的细粒度反馈**：apply 的逐资源失败分类/可重试维度较弱，且单一 IaC 不解决应用层操作的机器可读结果。这是「单标准跨三公理却仍留空格」的典型例证。

- **SemVer** —— 主投影 **①**，且**仅命中一条**。`MAJOR.MINOR.PATCH` 把兼容性编码进版本号，精确对应 ①「版本兼容性 MUST 机器可判定（…加=兼容 / 改=破坏）」。**不覆盖 ②③④**：它是纯版本语义约定，不涉契约内容、反馈、状态或边界——最能说明「一个标准只点亮一格」。

- **OpenTelemetry / 结构化日志** —— 主投影 **②**。结构化 span/event/属性使每个操作产出机器可读结果，命中 ②「每操作 MUST 返回机器可读结果」；带 `status.code` 与错误属性命中「失败 MUST 机器可分类」，错误类型/可重试语义触及「失败分类 SHOULD 含错误域+可重试」；trace 上下文与资源属性可承载 ②「产物 SHOULD 携带 provenance」。**不覆盖 ①③④**：它**观测**行为但不**定义契约**、不外置可重入状态、不设安全边界。

- **SLSA provenance** —— 主投影 **②**，且**仅命中 provenance 一条**。in-toto/SLSA 出处证明记录构建的源、依赖、构建者与参数，精确对应 ②「产物 SHOULD 携带 provenance（输入指纹+代码版本+执行者+时间）」+「构建与产出 SHOULD 可复现」。**不覆盖 ①③④**：它证明产物来源，但不提供接口契约、运行期反馈、状态恢复或操作边界。

### 4.3 结论：互相独立，必须组合

上表每一行都留有空格——没有任何单一标准填满四列：

- **MCP** 是当前最接近 L2 接口层的事实标准（同时投影 ①②），但对 ③④ 完全沉默；
- **OpenAPI / SemVer / llms.txt / AGENTS.md** 各只点亮 ① 的一到两条；
- **OTel / SLSA** 各只点亮 ② 的部分条目；
- **IaC** 罕见地横跨 ①③④，却恰好缺 ② 的细粒度反馈。

因此**四公理是互相独立的设计维度，而非某标准的不同切面**。要达 L3，必须**组合**：例如「OpenAPI/MCP（①）+ OTel+SLSA（②）+ IaC/事务（③）+ 沙箱+plan/approve（④）」。本规范的价值正在于提供这一**正交坐标系**，使任意标准栈的覆盖面与缺口可被机器化盘点。

## 附录 A · 跨项目类型 × 四力对照表

证明四公理与项目类型无关——每格是该象限的**具体落地**：

| 项目类型 | ① 确定性契约 | ② 可验证闭环 | ③ 可恢复状态 | ④ 安全边界 |
| --- | --- | --- | --- | --- |
| 库 / SDK | 完整类型签名 + SemVer + 机器可读 API spec | 结构化异常(带 code) + 确定性 | 纯函数 / 无隐藏全局态 | 废弃有迁移路径、无破坏性默认 |
| 服务 / API | OpenAPI + 错误码枚举 + 版本协商 | 结构化日志 + trace + 健康检查 | 幂等键 + 事务 + 可重放 | 写操作确认 / 限流 / 回滚 |
| 前端 | 类型化 props + 设计 token + ARIA 语义 | 可断言状态 + 错误边界 | 可恢复表单 + 乐观更新回滚 | 危险动作二次确认 + 撤销 |
| CLI | `--help` 机器可读 + 子命令 schema | `--json` 输出 + 语义化退出码 | 可重入 + 检查点 | `--dry-run`，危险需 `--force` |
| 数据产线 | schema 文件 + manifest 发现入口 | 校验门禁 + 数据指纹 + provenance | 增量 + 缓存 + 断点续跑 | 发版确认 + 版本不可变 |
| 基础设施 / IaC | 声明式期望状态 | plan + 状态可查 | apply 幂等收敛 | plan 预览 + 审批 + 回滚 |

## 一句话

> **单一真源 → 自动派生 → 自动校验锁一致 → 机器可读契约暴露 → 行动闭环反馈 → 状态可恢复 → 危险操作设界。**
> 对任何 Agent 产 / 消的软件都成立。

*本规范与任何具体项目解耦。欢迎作为独立标准演进。*
