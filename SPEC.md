# Agent-Native 软件设计规范

> **Agent-Native Software Specification**
> 版本 0.1 · 草案 · 2026-06-07 · 技术栈无关 · 领域无关

## 摘要

当软件的**开发者与消费者都是 AI Agent**而非人类，其「接口面」必须为 Agent 的认知特性重建。
本规范定义四条公理——**确定性契约、可验证闭环、可恢复状态、安全边界**——与一致性验收方法，
适用于任何软件产物（库 / 服务 / 前端 / CLI / 数据 / 基础设施），不绑定任何语言、框架或领域。

本文档中关键词「**必须**(MUST)」「**应当**(SHOULD)」「**可以**(MAY)」依 RFC 2119 解释。

---

## 0. 动机

人类与 Agent 的认知差异是**固定的**，软件接口若为人类优化，对 Agent 即为歧义与失明之源：

| | 人类 | AI Agent | 推导出的公理 |
| --- | --- | --- | --- |
| 记忆 | 持久、有经验 | 上下文有限且易失 | ③ 可恢复状态 |
| 歧义 | 直觉脑补 | 要么有契约，要么幻觉 | ① 确定性契约 |
| 行动 | 直接操作 | 经工具行动，需反馈才知对错 | ② 可验证闭环 |
| 试错 | 成本高、串行 | 成本低、可并行；**不可逆操作例外** | ④ 安全边界 |
| 阅读 | 叙事 | 结构 | ① 确定性契约 |

Agent-native 不是某类项目的模板，而是这组差异推导出的、跨领域的设计公理。

## 1. 范围

适用于任何被 Agent 开发或消费的软件产物，与具体架构正交。一个项目可只满足部分公理；
满足度即其 **agent-native 成熟度**（见 §3）。本规范描述**做什么**与**为何**，不规定实现技术。

## 2. 四公理（规范性）

### 公理一 · 确定性契约 — Determinism over Inference

**定义**：凡需 Agent 推断之处皆为幻觉源。一切接口、数据、版本、能力**必须**机器可判定、自描述、单一真源。

**规范要求**
- 接口契约 **MUST** 机器可读（类型签名 / JSON Schema / OpenAPI / IDL），不得**仅**以散文表达。
- 同一事实 **MUST** 单一真源；派生物自动生成，并由自动校验锁定一致（防漂移）。
- 版本兼容性 **MUST** 机器可判定（语义化版本 + 显式兼容语义：加 = 兼容 / 改 = 破坏）。
- 项目能力与入口 **SHOULD** 有机器可读的发现清单，Agent 无需读散文或猜目录结构。
- 面向人的叙事文档与面向 Agent 的契约 **SHOULD** 分离（各自最优）。

**反模式**：散文即契约、隐式默认、魔法常量、靠经验的约定俗成。

### 公理二 · 可验证闭环 — Closed-Loop Verifiability

**定义**：Agent 经工具行动，**必须**从行动获得机器可读的结果与失败原因，方能自我纠错。沉默的成功与散文的失败 = Agent 失明。

**规范要求**
- 每个操作 **MUST** 返回机器可读结果（结构化输出 + 语义化退出码 / 状态）。
- 失败 **MUST** 机器可分类（错误码 / 类型），不得**仅**为自然语言消息。
- 产物 **SHOULD** 携带 provenance（输入指纹 + 代码版本 + 执行者 + 时间），可被另一 Agent 独立复现与验证。
- 构建与产出 **SHOULD** 可复现（确定性）。

**反模式**：成功静默、失败 print 散文、副作用不可观测、产出不可复现。

### 公理三 · 可恢复状态 — Resumable State

**定义**：Agent 上下文易失、长任务会中断。状态**必须**外置、可重入。

**规范要求**
- 长流程 **MUST** 幂等或可断点续跑；中断后重跑不得产生重复副作用或损坏数据。
- 状态 **MUST** 外置于持久载体，不得**仅**存于 Agent 上下文或隐藏的全局可变态。
- 操作 **SHOULD** 幂等（幂等键 / 期望状态收敛）。

**反模式**：一次性不可重入脚本、隐藏可变状态、"挂了从头来"。

### 公理四 · 安全边界 — Safe-by-Default Boundaries

**定义**：Agent 行动快且会错。可逆操作可自由试错；**不可逆 / 外发 / 破坏性操作必须设界**。

**规范要求**
- 不可逆 / 外发 / 破坏性操作 **MUST** 显式确认或具备回滚，不得为静默默认。
- 操作 **SHOULD** 提供 dry-run / preview。
- Agent 运行 **SHOULD** 遵循最小权限，且变更可审计、可回滚。

**反模式**：破坏性默认、无 dry-run、无回滚、宽权限、不可审计。

## 3. 一致性（Conformance）

### 3.1 四问验收（机械可执行）

对任意项目，逐条回答；全为「是」即符合本规范：

1. **契约**：一个从未见过它的 Agent，能否**只靠机器可读文件**（不读散文、不问人）正确调用 / 消费 / 扩展它？
2. **反馈**：执行任一操作后，能否**程序化判断**成功 / 失败 / 失败原因？
3. **恢复**：任一长流程中断后重跑，是否**安全且不重复副作用**？
4. **安全**：Agent 能否在**不造成不可逆损害**的前提下自由试错？

### 3.2 成熟度分级

| 级别 | 名称 | 条件 | Agent 可做到 |
| --- | --- | --- | --- |
| **L0** | 人类原生 | 散文契约、人读日志、不可逆默认 | 需人类在环翻译 |
| **L1** | 契约就绪 | 满足公理一 | 正确消费 / 扩展，无需猜 |
| **L2** | 闭环就绪 | + 公理二、三 | 自我纠错、中断可恢复 |
| **L3** | 自治就绪 | + 公理四 | 在安全边界内自主开发与运维 |

## 4. 与现有工作的关系

本规范不发明新机制，而是把已有实践统一进四力坐标系：

- **公理一**：类型系统、JSON Schema / OpenAPI / Protobuf、SemVer、llms.txt、AGENTS.md、声明式 IaC（期望状态）。
- **公理二**：结构化日志 / OpenTelemetry、MCP 工具结果、确定性构建、SLSA provenance。
- **公理三**：幂等 API、事务、IaC 收敛、可 resume 的工作流编排。
- **公理四**：IaC plan/approve、dry-run 惯例、最小权限 / 能力沙箱。

MCP 同时投影到公理一（能力契约）与公理二（工具调用反馈），是当前最接近 L2 接口层的事实标准。

## 附录 A · 跨项目类型 × 四力对照表

证明四公理与项目类型无关——每格是该象限的**具体落地**：

| 项目类型 | ① 确定性契约 | ② 可验证闭环 | ③ 可恢复状态 | ④ 安全边界 |
| --- | --- | --- | --- | --- |
| 库 / SDK | 完整类型签名 + SemVer + 机器可读 API spec | 结构化异常(带 code) + 确定性 | 纯函数 / 无隐藏全局态 | 废弃有迁移路径、无破坏性默认 |
| 服务 / API | OpenAPI + 错误码枚举 + 版本协商 | 结构化日志 + trace + 健康检查 | 幂等键 + 事务 + 可重放 | 写操作确认 / 限流 / 回滚 |
| 前端 | 类型化 props + 设计 token + ARIA 语义 | 可断言状态 + 错误边界 | 可恢复表单 + 乐观更新回滚 | 危险动作二次确认 + 撤销 |
| CLI | `--help` 机器可读 + 子命令 schema | `--json` 输出 + 语义化退出码 | 可重入 + 检查点 | `--dry-run`，危险需 `--force` |
| 数据产线 | schema 文件 + manifest 发现入口 | 校验门禁 + 数据指纹 + provenance | 增量 + 缓存 + 断点续跑 | 发版确认 + 版本不可变 |
| 基础设施 / IaC | 声明式期望状态 | plan + 状态可查 | apply 幂等收敛 | plan 预览 + 审批 + 回滚 |

---

## 一句话

> **单一真源 → 自动派生 → 自动校验锁一致 → 机器可读契约暴露 → 行动闭环反馈 → 状态可恢复 → 危险操作设界。**
> 对任何 Agent 产 / 消的软件都成立。

*本规范与任何具体项目解耦。欢迎作为独立标准演进。*
